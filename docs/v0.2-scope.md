# School Climate Hub — v0.2 Scope

**Status:** draft · 2026-05-14
**Owner:** Reza Malik (BT) · Premier DLC (Erum)
**Predecessor:** v0.1 (static HTML, anonymous chat, UNICEF Venture Fund submission)

This document is the build spec for v0.2. v0.1 is preserved on branch `legacy/static-html-v0.1` and remains live for the UNICEF review window.

---

## 1. Goal

Turn `schoolclimatehub.org` from a single-tenant prototype dashboard into a multi-tenant web app that the public can use to monitor school-level climate risk and subscribe to alerts, that operators can use to manage their schools, and that scales without burning a budget on infrastructure or AI credits.

Specifically:

1. Public can browse, search, and subscribe to alerts for any of the 50 pilot schools — no account required.
2. Operators can claim and manage their school, with verification.
3. Free notifications via **web push (PWA)**. Email deferred to v0.3.
4. AI chat behind sign-in with per-user rate limits — eliminates uncapped API spend.
5. Multilingual (EN, UR, AR, FR) with RTL support.
6. Built on a stack that runs at ~$20/month at pilot scale.

---

## 2. Audience & roles

| Role | Sign-in | Can do |
|---|---|---|
| **Public visitor** | No | Browse data, search schools, view forecasts, read docs |
| **Public subscriber** | No (email + signed unsubscribe tokens) | Subscribe to alerts for up to 10 schools, manage preferences via tokenised links |
| **Operator** | Yes (magic link) | Subscriber capabilities + dispatch advisories, manage own school's thresholds and roster, view subscriber counts (never addresses) |
| **District admin** | Yes | Operator capabilities × N schools in their district |
| **Reviewer** (UNICEF / BT / PDLC) | Yes | Read-only across all data; can chat with AI |

Public subscribers do **not** have accounts. This keeps friction low, attack surface small, and privacy posture clean. They manage prefs via signed token URLs embedded in every notification.

AI chat is behind sign-in for cost control. The general public cannot trigger Claude calls.

## 4. Information architecture

The app explains itself — no separate marketing landing page.

### Top bar (always visible)
- Logo → home
- Search (school name / EMIS) — fuzzy, jumps to school detail
- Theme picker (light / system / dark)
- Language switcher (EN / UR / AR / FR; RTL for UR + AR)
- Help menu (About, Methodology, Data sources, API, Privacy, For operators, Contact)
- Account menu (Sign in for operators; "Subscribe" CTA otherwise)

### First-visit onboarding
Three dismissible cards at the top of `/`:
1. What is this? (one sentence + a stat row)
2. Where does the data come from? (ERA5 / MODIS / ECMWF, refresh cadence)
3. How to subscribe (email + push, no account needed)

Dismissal persists in localStorage.

### Left nav
| Item | Public | Subscriber | Operator | Admin | Reviewer |
|---|---|---|---|---|---|
| Overview (map + alerts + clusters) | ✓ | ✓ | ✓ | ✓ | ✓ |
| Schools (RAG list) | ✓ | ✓ | ✓ | ✓ | ✓ |
| My Schools (subscriptions) | — | ✓ | ✓ | ✓ | — |
| Open Data (API + downloads) | ✓ | ✓ | ✓ | ✓ | ✓ |
| Alerts (history) | — | ✓ | ✓ | ✓ | ✓ |
| Dispatch (compose & send) | — | — | ✓ | ✓ | — |
| Admin (claims, users) | — | — | — | ✓ | — |
| Settings | thresholds (view) | + prefs | + school | + district | view |

### Screen-by-screen surface

**Overview** — map with markers, today's alerts (real, from current scores + thresholds + forecast), cluster roll-up, command actions (operator-only: broadcast advisory, modify school day, export PDF, acknowledge).

**Schools** — RAG table, filter by cluster/hazard/score, click row → school detail. Public users see a "Subscribe" button per row.

**School detail** — metadata, current scores, raw values, **15-day forecast sparkline** (HRES line + ENS p10–p90 band + P(red) overlay), advisory history, subscriber count (operator), "Ask Claude for detail" (auth-gated).

**Open Data** — real downloads, OpenAPI spec for `/api/v1/schools|scores|forecast`, schema docs with field definitions/units/sources, last-update timestamps.

**My Schools** (subscriber) — list of subscriptions, per-school preferences toggle, unsubscribe.

**Alerts** (signed in) — feed of received alerts, mark read, filter by school/hazard.

**Dispatch** (operator) — compose advisory, AI-draft (EN/UR/PA), approve and fan out via push.

**Admin** (admin) — claim queue, user list, district roll-up.

**Settings** — role-aware (see role table above).

## 5. Flows

### Subscriber sign-up
1. Home → "Subscribe for your school's alerts" CTA.
2. School picker (search by name or EMIS, multi-select up to 10).
3. Enter email → Turnstile → submit.
4. "Check your inbox" screen.
5. Magic-link arrives (Resend transactional, from `auth.schoolclimatehub.org`).
6. Click link → email verified, subscriptions saved, signed in temporarily.
7. Browser push permission prompt.
8. Done — receives alerts when subscribed school crosses thresholds.

All automated. No human in the loop.

### Operator school claim
1. Sign in (magic link, any role starts as subscriber).
2. Settings → "Claim a school I manage" → search → select.
3. Submit claim with role + title + evidence.
4. Trust hierarchy:
   - **A. Verified roster match** (email on PDLC-supplied roster) → instant approval.
   - **B. Domain match** (email on `*.edu.pk`, `*.gov.pk` allowlist) → instant approval.
   - **C. Manual review** → "pending" badge, BT/PDLC admin notified → approve/reject via admin panel.
5. On approval: role upgrades to operator, notification email.

PDLC supplies the verified-roster CSV once for the 50 pilot schools (~10 min of their time). Without it, path C handles all claims (low volume, low cost).

### Subscribe / unsubscribe / manage prefs
- Subscribe: "Subscribe" button on a school → if not yet signed up, pivot to sign-up flow; if already a subscriber, instant.
- Unsubscribe: from "My Schools" toggle, from notification deep link, or via "delete account" cascade.
- Preferences (per subscription): web push on/off, severity filter (red only / amber+red / all), quiet hours (default 22:00–06:00 local).
- All editable without sign-in via signed token URLs in every notification.

### Operator dispatch
1. Operator views their school's detail.
2. Clicks "Send advisory".
3. AI drafts EN/UR/PA text from current scores + forecast.
4. Operator edits and clicks Approve.
5. Cron worker fans out to subscribers (web push, respecting quiet hours and per-user prefs).
6. Advisory recorded in D1, visible in subscriber Alerts feed.

The "send" decision is always human. The drafting and fan-out are automated.

### Automatic threshold-crossing alerts
- Cron worker runs hourly, evaluates current scores + 24h forecast vs thresholds.
- Any new red-threshold crossing triggers a notification to all subscribers of that school.
- These are informational; no operator approval (rate-limited to 1 per school per day, with a "still active" suppression).

## 6. What stays manual (by design)

- **Operator verification fallback** — when neither roster nor domain match (path C).
- **Approve & dispatch advisory** — operator is accountable; must click Approve.
- **Per-school threshold tuning** — operators can override defaults with local knowledge.
- **Translation review of dispatched advisories** — AI drafts, operator approves wording before fan-out.
- **PDLC roster ingestion** — one-time CSV upload, then trickle.
- **Bug reports / support** — email / form to BT team.

Everything else is automated.

## 7. Limits (anti-abuse, capacity, deliverability)

| Limit | Default |
|---|---|
| Subscribers per school | 500 during pilot (waitlist after) |
| Schools per subscriber | 10 |
| Sign-ups per IP per day | 5 |
| Magic-link requests per email | 1/min, 5/hr |
| Push notifications per subscriber per day | 5, with cross-school batching |
| Chat queries per signed-in user per day | 30 |
| Turnstile on all public forms | Always on |

Configurable. Start strict; loosen if real demand outpaces.

## 8. Email deliverability (magic links)

Email is **only** used in v0.2 for magic-link auth. No marketing, no welcome, no "we miss you", no alerts. Lowest-risk email category there is.

Posture:
1. **Validate before send** — regex + MX record lookup at form submission.
2. **Dedicated sub-domain** — `auth.schoolclimatehub.org` for transactional; SPF/DKIM/DMARC on the sub-domain. Reputation isolated from apex.
3. **Send only what user requested** — one email per sign-in attempt.
4. **Hard-bounce → permanent suppression** in D1 and at the ESP.
5. **Resend free tier** (3k/mo) covers pilot volume.
6. **Google Postmaster Tools** + **Microsoft SNDS** for reputation monitoring.

Alert emails are deferred to v0.3 with a separate sub-domain (`alerts.schoolclimatehub.org`) and a proper warm-up plan.

## 9. Out of scope for v0.2 (explicit)

- Email alert notifications → v0.3 (own sub-domain, own deliverability warm-up)
- Native mobile app → never (PWA covers it)
- WhatsApp / SMS → v0.3+ (paid channels)
- Multi-tenant beyond Gujranwala-50 → v0.4 (UNICEF country offices, MoE partnerships)
- Custom advisory templates per district → v0.4
- Native chat for subscribers (only operators+ have chat in v0.2)
- Per-school custom branding
- Federated trust via national EMIS APIs → v0.5+

## 12. Success criteria for v0.2

- Public can sign up and receive a real alert end-to-end within the pilot 50.
- An operator can claim a school via path A or B without manual review.
- An operator can dispatch a parent advisory in their language of choice.
- Chat is gated; non-authenticated traffic to `/api/chat` returns 401.
- Total monthly cost stays under $25.
- All copy is available in EN and UR at minimum; AR + FR within the same release window.
- Privacy posture documented and counsel-approved before launch.
